## 1. 项目背景

### 1.1 项目概述

StockView 是一款面向 A 股中短线投资者的专业看盘工作台，覆盖实时行情监控、技术分析、资金流向追踪、智能选股等核心交易决策场景。当前前端已完成纯浏览器端 SPA 实现，需配套建设后端服务以支撑数据聚合、缓存加速、用户系统、实时推送等能力。

### 1.2 当前架构

```
┌───────────────────────────────────────────────────┐
│                  浏览器 (Frontend)                  │
│  index.html / login.html / ECharts / JS 逻辑       │
│  ┌───────────────────────────────────────────────┐ │
│  │  LocalStorage (用户/自选股)                    │ │
│  └───────────────────────────────────────────────┘ │
└─────────────────────┬─────────────────────────────┘
                      │ JSONP 跨域
         ┌────────────┴────────────┐
         ▼                        ▼
 ┌──────────────┐    ┌──────────────────────┐
 │ 东方财富 API  │    │ Mock 数据 (离线兜底)   │
 └──────────────┘    └──────────────────────┘
```

**存在的问题**：
- JSONP 跨域请求存在安全隐患和超时风险
- 无后端缓存，每次页面刷新重复请求
- 用户数据仅存 LocalStorage，多设备无法同步
- 实时行情需轮询，无法 WebSocket 主动推送
- API Token（如 TSDP Token）暴露在前端

## 2. 技术方案对比

### 2.1 候选方案概览

| 维度 | 方案一：Next.js App Router 🏆 | 方案二：Python FastAPI | 方案三：Go + Gin |
|------|------------------------------|----------------------|-----------------|
| **开发语言** | 全栈 TypeScript (前后端统一) | Python | Go |
| **团队门槛** | 低（前端可直接参与） | 中（需双语言维护） | 高（新语言栈） |
| **开发效率** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **运行性能** | ⭐⭐⭐⭐ (RSC + 流式) | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| **实时推送** | ⭐⭐⭐⭐ (自定义 Server + WS) | ⭐⭐⭐ (需 ASGI) | ⭐⭐⭐⭐⭐ |
| **金融数据生态** | ⭐⭐⭐ (axios HTTP) | ⭐⭐⭐⭐⭐ (sxsc_tushare SDK) | ⭐⭐ |
| **部署复杂度** | ⭐⭐⭐⭐⭐ (Vercel / Node Server) | ⭐⭐⭐⭐ (需 Python env) | ⭐⭐⭐⭐⭐ (单二进制) |
| **团队友好度** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |
| **适合场景** | 全栈 JS 项目 + SSR | 数据科学 + 量化 | 高并发中间件 |

### 2.2 推荐方案：Next.js App Router

选择 **Next.js App Router** 的核心理由：

1. **前后端统一为 TypeScript**：前端代码、后端 API、数据模型共享同一套类型定义，杜绝前后端数据契约不一致。
2. **App Router 文件即路由**：`app/api/stocks/route.ts` 天然对应 `/api/stocks`，零配置路由注册，目录结构即 API 文档。
3. **React Server Components 潜力**：后续如需 SSR 页面（大盘概览、排行榜），RSC 可直接在服务端聚合数据，减少客户端请求。
4. **增量迁移友好**：前端 SPA 页面可以直接 `app/page.tsx` 逐步接管，不需要一次性重写。
5. **生态与部署**：Vercel 一键部署；也可 `next start` 以传统 Node.js Server 模式运行，搭配 WebSocket 自定义服务器。
6. **内置优化**：Image 优化、 Streaming、缓存策略（`stale-while-revalidate`）开箱即用。

#### 与 Express 方案对比

| 对比维度 | Express | Next.js App Router |
|---------|---------|-------------------|
| 路由定义 | `router.get('/api/stocks', handler)` | `app/api/stocks/route.ts` 文件约定 |
| 中间件 | `app.use(middleware)` | `middleware.ts` 单文件 + Route Handler 内组合 |
| 类型安全 | 无原生支持 | `NextRequest` / `NextResponse` 全类型 |
| API 文档 | 需单独维护 | 目录结构即文档 |
| 静态资源 | `express.static` | `public/` 文件夹 |
| WebSocket | `ws` 库 + `server.on('upgrade')` | 自定义 server.js 同方案 |
| 学习成本 | 低但无约束 | 中（约定优于配置） |
| 部署选项 | 单一 Node.js | Vercel / Node.js / Docker |

#### 架构说明

Next.js 默认以无状态 Serverless 模式运行（部署到 Vercel），但我们的场景需要：

1. **WebSocket 长连接** → 需要自定义 Node.js Server 持有状态
2. **内存缓存** → 需要进程内常驻内存
3. **定时任务** → 需要进程级 cron 调度

因此部署方式为 **Node.js Server 模式**（`server.ts` 手动启动），使用 Next.js 的 `requestHandler` 将 API 路由托管给 App Router，同时附加 WebSocket 服务。

```
Next.js App Router 提供 : Route Handlers + Middleware
自定义 Server 提供    : WebSocket 升级 + 全局进程生命周期
SQLite 提供           : 持久化存储
```

### 4.3 核心代码示例

#### 4.3.1 自定义服务器入口 (`server.ts`)

这是整个后端的启动入口，将 Next.js App Router 与 WebSocket 整合到同一进程：

```typescript
import { createServer } from 'http';
import { parse } from 'url';
import next from 'next';
import { initWebSocket } from '@/ws';
import { initScheduler } from '@/jobs/scheduler';

const dev = process.env.NODE_ENV !== 'production';
const app = next({ dev });
const handle = app.getRequestHandler();

app.prepare().then(() => {
  const server = createServer((req, res) => {
    const parsedUrl = parse(req.url!, true);
    handle(req, res, parsedUrl);  // 所有 Next.js App Router 路由由此处理
  });

  // 在同一个 HTTP 服务器上附加 WebSocket
  initWebSocket(server);

  // 启动定时任务
  initScheduler();

  const port = process.env.PORT || 3000;
  server.listen(port, () => {
    console.log(`> StockView Server ready on http://localhost:${port}`);
  });
});
```

#### 4.3.2 Route Handler 示例 (`app/api/stocks/route.ts`)

```typescript
import { NextRequest, NextResponse } from 'next/server';
import { getQuotes } from '@/lib/services/market';
import { memoryCache } from '@/lib/cache';

export async function GET(request: NextRequest) {
  const codes = request.nextUrl.searchParams.get('codes');
  if (!codes) {
    return NextResponse.json(
      { code: 40001, msg: '缺少 codes 参数' },
      { status: 400 }
    );
  }

  const codeList = codes.split(',').map(s => s.trim());

  // 尝试读缓存
  const cacheKey = `quotes:${codeList.join(',')}`;
  const cached = memoryCache.get(cacheKey);
  if (cached) {
    return NextResponse.json({
      code: 0,
      data: cached,
      meta: { cached: true, timestamp: new Date().toISOString() }
    });
  }

  try {
    const data = await getQuotes(codeList);
    memoryCache.set(cacheKey, data, 1000); // 1秒缓存
    return NextResponse.json({
      code: 0,
      data,
      meta: { cached: false, timestamp: new Date().toISOString() }
    });
  } catch (error) {
    return NextResponse.json(
      { code: 50001, msg: '数据获取失败', detail: (error as Error).message },
      { status: 500 }
    );
  }
}
```

#### 4.3.3 中间件 (`middleware.ts`)

```typescript
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

const rateLimitMap = new Map<string, { count: number; resetAt: number }>();

export function middleware(request: NextRequest) {
  // 仅处理 API 路由
  if (!request.nextUrl.pathname.startsWith('/api')) {
    return NextResponse.next();
  }

  // CORS 头
  const response = NextResponse.next();
  response.headers.set('Access-Control-Allow-Origin', '*');
  response.headers.set('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE, OPTIONS');
  response.headers.set('Access-Control-Allow-Headers', 'Content-Type, Authorization');

  // Preflight 直接返回
  if (request.method === 'OPTIONS') {
    return new NextResponse(null, { status: 204 });
  }

  // 简易 IP 限流 (100 req/min)
  const ip = request.headers.get('x-forwarded-for') || 'unknown';
  const now = Date.now();
  const entry = rateLimitMap.get(ip);

  if (entry && now < entry.resetAt) {
    entry.count++;
    if (entry.count > 100) {
      return NextResponse.json(
        { code: 42901, msg: '请求过于频繁，请稍后再试' },
        { status: 429 }
      );
    }
  } else {
    rateLimitMap.set(ip, { count: 1, resetAt: now + 60000 });
  }

  return response;
}
```

#### 4.4.3 WebSocket 服务实现 (`ws/index.ts`)

```typescript
import { Server as HttpServer } from 'http';
import { WebSocketServer, WebSocket } from 'ws';
import { fetchQuotes } from '@/lib/services/market';

interface ClientState {
  ws: WebSocket;
  subscribedCodes: Set<string>;
  alive: boolean;
}

const clients = new Map<WebSocket, ClientState>();

export function initWebSocket(server: HttpServer) {
  const wss = new WebSocketServer({ server, path: '/ws' });

  wss.on('connection', (ws) => {
    const state: ClientState = { ws, subscribedCodes: new Set(), alive: true };
    clients.set(ws, state);

    ws.on('message', (raw) => {
      try {
        const msg = JSON.parse(raw.toString());
        if (msg.type === 'subscribe') {
          msg.codes.forEach((code: string) => state.subscribedCodes.add(code));
        } else if (msg.type === 'unsubscribe') {
          msg.codes.forEach((code: string) => state.subscribedCodes.delete(code));
        } else if (msg.type === 'ping') {
          ws.send(JSON.stringify({ type: 'pong' }));
        }
      } catch { /* ignore malformed */ }
    });

    ws.on('close', () => {
      clients.delete(ws);
    });

    ws.on('pong', () => { state.alive = true; });
  });

  // 心跳检测
  const heartbeat = setInterval(() => {
    wss.clients.forEach((ws) => {
      const state = clients.get(ws);
      if (state && !state.alive) {
        ws.terminate();
        clients.delete(ws);
        return;
      }
      if (state) {
        state.alive = false;
        ws.ping();
      }
    });
  }, 30000);

  wss.on('close', () => clearInterval(heartbeat));

  // 行情推送循环
  startPushLoop();
}

async function startPushLoop() {
  const TICK_INTERVAL = 1000; // 每秒推送

  setInterval(async () => {
    // 收集所有订阅的股票代码
    const allCodes = new Set<string>();
    clients.forEach((state) => {
      state.subscribedCodes.forEach(code => allCodes.add(code));
    });

    if (allCodes.size === 0) return;

    try {
      const quotes = await fetchQuotes(Array.from(allCodes));
      const message = JSON.stringify({ type: 'tick', data: quotes });

      clients.forEach((state) => {
        if (state.ws.readyState === WebSocket.OPEN) {
          state.ws.send(message);
        }
      });
    } catch {
      // 静默失败，下次重试
    }
  }, TICK_INTERVAL);
}
```

```typescript
async function macdGoldenStrategy(tradeDate: string): Promise<StrategyResult[]> {
  const stocks = await tushare.stockBasic('L');
  const basics = await tushare.dailyBasic(tradeDate);
  const basicMap = new Map(basics.map(b => [b.ts_code, b]));

  const results: StrategyResult[] = [];

  for (const stock of stocks.slice(0, 200)) { // 分批处理，避免超时
    const kline = await tushare.daily(stock.ts_code, '', tradeDate);
    if (kline.length < 26) continue;

    const macd = calcMACD(kline.map(k => k.close));
    const latest = macd[macd.length - 1];
    const prev = macd[macd.length - 2];

    if (prev.dif > prev.dea || latest.dif <= latest.dea) continue;
    if (latest.hist <= 0) continue;

    const basic = basicMap.get(stock.ts_code);
    if (!basic || !basic.volume_ratio || basic.volume_ratio < 1.0 || basic.volume_ratio > 2.5) continue;

    results.push({
      ts_code: stock.ts_code,
      name: stock.name,
      price: kline[kline.length - 1].close,
      change: 0,
      signals: ['MACD金叉', '温和放量'],
      score: basic.volume_ratio * 10,
    });
  }

  return results.sort((a, b) => b.score - a.score);
}

  strategyId: StrategyId,
  tradeDate: string

  const stocks = await macdGoldenStrategy(tradeDate); // 实际根据 strategyId 分发
  return { total: stocks.length, stocks };
}
```

### 4.7 缓存层设计

```typescript
// lib/cache/memory.ts

interface CacheEntry<T> {
  data: T;
  expiresAt: number;
}

export class MemoryCache {
  private store = new Map<string, CacheEntry<any>>();
  private hitCount = 0;
  private missCount = 0;

  get<T>(key: string): T | null {
    const entry = this.store.get(key);
    if (!entry) { this.missCount++; return null; }
    if (Date.now() > entry.expiresAt) {
      this.store.delete(key);
      this.missCount++;
      return null;
    }
    this.hitCount++;
    return entry.data as T;
  }

  set<T>(key: string, data: T, ttlMs: number): void {
    this.store.set(key, { data, expiresAt: Date.now() + ttlMs });
  }

  /** 缓存命中率监控 */
  stats() {
    const total = this.hitCount + this.missCount;
    return {
      size: this.store.size,
      hits: this.hitCount,
      misses: this.missCount,
      hitRate: total > 0 ? (this.hitCount / total * 100).toFixed(1) + '%' : '0%',
    };
  }
}

// 全局单例
export const memoryCache = new MemoryCache();
```

## 6. 部署方案

### 6.1 环境要求

| 资源 | 最低配置 | 推荐配置 |
|------|---------|---------|
| CPU | 1 核 | 2 核 |
| 内存 | 1 GB | 4 GB |
| 磁盘 | 10 GB | 20 GB SSD |
| 操作系统 | Linux / macOS / Windows | Linux (Ubuntu 22.04) |
| Node.js | >= 18 LTS | >= 20 LTS |

### 6.2 部署架构

```
                        ┌──────────────┐
                        │   Nginx 443   │
                        │  HTTPS 转发    │
                        │  /ws  → WS    │
                        └──────┬───────┘
                               │
                        ┌──────▼───────┐
                        │  PM2 Process │
                        │  server.ts   │
                        │  :3000       │
                        └──────┬───────┘
                               │
                        ┌──────▼───────┐
                        │  Next.js      │
                        │  App Router   │
                        │  + WebSocket  │
                        │  + SQLite     │
                        └──────────────┘
```

> **为什么不用 Vercel 部署？**  
> 因为需要 WebSocket 长连接 + 内存缓存 + 定时任务（node-cron）+ SQLite 文件数据库，这些都需要常驻进程。Vercel Serverless 环境无法满足这些需求。  
> 如未来需要扩展到 Serverless + 托管数据库（如 PostgreSQL + Redis），可以平滑迁移，仅需调整 `lib/db/` 和 `ws/` 两层。

## 8. 风险评估

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| TSDP 接口限流或变更 | 数据无法获取 | 中 | SQLite 缓存兜底 + 东方财富 API 备用 |
| WebSocket 连接数过高 | 服务端资源耗尽 | 低 | 连接数上限 + 心跳断开僵尸连接 |
| 选股器全市场计算慢 | 接口响应超时 | 中 | 分批处理 + 后台预计算 + 结果缓存 5min |
| Next.js 版本更新破坏兼容 | 构建失败 | 低 | 锁定 `next` 版本，CI 先行验证 |
| SQLite 并发写冲突 | 数据异常 | 低 | WAL 模式 + 事务重试机制 |
| 自定义 Server 模式脱离 Vercel 生态 | 无法享受 Serverless 红利 | 低 | 架构上做好服务层抽象，未来可拆分 |

## 9. 总结与建议

### 9.1 推荐结论

**采用 Next.js App Router + 自定义 Node.js Server**，理由如下：

1. **全栈 TypeScript** — 前后端共享类型、工具函数、常量枚举，根除数据契约不一致问题
2. **App Router 约定式路由** — 目录结构即 API 文档，零路由配置，维护成本极低
3. **渐进式架构演进** — 当前 SPA 可直接放入 `public/` 由 Next.js 托管，后续逐步迁移为 SSR 页面
4. **WebSocket + REST 一体** — 自定义 Server 将 Next.js Route Handlers 与 WebSocket 整合在同一进程，架构简洁
5. **部署灵活** — 可 Docker 部署、PM2 管理、也保留了未来迁移到 Vercel + 托管数据库的路径

### 9.2 与 Express 方案的关键区别

| 维度 | Express | Next.js App Router ✅ |
|------|---------|----------------------|
| 路由定义 | `router.get('/api/stocks', handler)` | `app/api/stocks/route.ts` 文件约定 |
| 类型安全 | 手动定义 | `NextRequest`/`NextResponse` 全类型 + 共享类型 |
| API 文档 | 需 Swagger/手动维护 | 目录结构即文档 |
| 前端集成 | 完全分离 | 同项目、同仓库、可 SSR |
| 部署 | Node.js Server | 自定义 Server / Vercel 双模式 |
